# README Audit Checklist

## Accuracy

- Project purpose matches the repository
- Features are supported by files, code, examples, or tests
- Install commands match the package manager and lockfiles
- Usage examples match actual binaries, scripts, or APIs
- Configuration names match code and sample env files
- License section matches repository license files

## Completeness

- New users can understand what the project does quickly
- Setup has prerequisites and install steps
- Usage has at least one concrete example
- Development and test commands are present when contributors are expected
- Troubleshooting covers known setup or runtime issues when evidence exists

## Structure

- Important information appears before deep details
- Headings are predictable and easy to scan
- Empty or speculative sections are absent
- Long command explanations are replaced with code blocks where possible
- Repeated content is consolidated

## Style

- Prose is concise and fluent
- Lists avoid sentence-final periods where possible
- Icons and badges are relevant rather than decorative
- Claims avoid hype and unsupported superlatives
- Tone matches the project and existing documentation

## Audit Severity

Use high severity for issues that block installation, usage, or correct understanding

Use medium severity for stale or misleading content, missing important sections, or confusing structure

Use low severity for polish, wording, ordering, or optional improvements
